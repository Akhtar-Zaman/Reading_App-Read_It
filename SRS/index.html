1. Introduction

1.1 Purpose

The purpose of this software requirement specification(SRS) is to give an overall idea of what “Glare Trail” is about. The mentioned product is an application which takes power of the world wide web to provide an interactive reading environment for students. Moreover, for teachers, it will allow quick assignment dispersal as well as generate a report based on the time spent by students in individual sections of the assignments. This will allow the teacher to create interesting sections based on student preferences. The application will implement an AI based vision tracking system. For students, the application will provide reading mode with customizable environment for a comfortable reading experience.

1.2 Business Case and Relevance to Gartner’s Prediction

According to Gartner’s Top Strategic Predictions for 2018 and Beyond, artificial intelligence can successfully categorize images as well as humans at a fast rate. Our app will take advantage of this and utilize it to create the aforementioned eye-ball tracking mechanism.

1.3 Document Conventions

The document currently in view is in accordance with IEEE SRS writing convention. Important quotes and (or) notations are highlighted by BLUE color and the font used is Arial.

1.4 Intended Audience and Reading Suggestions

This document is created with the intention of fulfilling the following criteria:

  • For team’s use (including all members along with the project manager)
  • For marketing people in order to bring in potential clients
  • For log-keepers to smoothen updating process
  • For developers to help in impending updates and modifications
  • For testers to ensure standard debugging methods

1.5 Product Scope

  • The app will provide the parties involved to get a hassle-free solution.
  • The faculties will be able to disperse their assignments to the intended students easily.
  • The students will be able to view the assignments at their own pace and in their own customized way.
  • The user-experience/user-interface(UX/UI) will be clean, plain and smooth besides being intuitive.
  • The design of the website will be following RWD (Responsive Web Design) with adaptive display size facility.
  • IAM (Identity and Access Management) will enable users to login with their social accounts.
  • Deployment for multi-platform will allow users to use it on their favorite electronic gadgets.

1.6 References

The reference of IEEE SRS documentation was taken as format in creation of this document.

2. Overall Description

2.1 Product Perspective

There will be user friendly smartphone application and web application for the students. Student will get their reading tasks from their respectable teachers. This application will track how much tasks is being completed by the students and this report will be sent to their respective teachers along with their identification by selfie image of the students. The teacher web application will show the tasks report of every individuals

2.2 Product Functions

By using this “Read It” students can view their reading list and the appointed tasks. For the teacher members it has adding class, students and reading materials and in reading material it can be sent to the students individually or to the whole class.

2.3 User Classes and Characteristics

Class Characteristics
System Administrator Can login to create and delete users, assign role and modify class list, student list, rendered document and reading analytics.
Teachers Teacher will register, login, manage classes and mange assignment and view student’s information.
Students Students will register, login, view reading list and can choose which to read. Students will give permission of their smartphone’s resources.

2.4 Operating Environment

  • The both teacher and student website will run on most of the common browsers like Google Chome, Firefox, Edge.
  • there will be an android app only for the students.

2.5 Design and Implementation Constraints

  • Product must be ready for delivery by deadline
  • This document must be followed srictly by the designers and developers

2.6 User Documentation

  • User will get this documentation with the application
  • There will be customer support through email

2.7 Assumptions and Dependencies

  • User has to connect to internet to upload reading information
  • Server must be always active

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  • User must login with Federated ID Provider.
  • Document has to be sent less than few seconds.
  • When teachars asked for the student report it has to be auto-generated in few seconds.

5.2 Safety Requirements

  • Smartphone application and website must follow proper security protocol like SSL, TLS and robust encryption like AES-256, Sha-512.
  • We will remove the background of the images prior to generating report for the teachers, respecting the privacy of students.
  • Our architecture does not support the raw image (with background) to be stored in the server.

5.3 Security Requirements

  • Website and smartphone application must create session a properly.

5.4 Software Quality Attributes

  • User friendly interface.
  • Students can change reading environment.
  • How much is being read will be can calculated based on eye tracking data.
  • User will be able to manage assignments and classes.

5.5 Business Rules

  • Users must have the email account of their respective organization.
  • Students must be able to select what file to read.
  • Teachars going to be able to mange classes and assignments.
